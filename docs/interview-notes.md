# 面试讲法

## 这个项目是做什么的

这是一个 C++ 小型服务项目，主要模拟游戏服务端里比较常见的三类模块：

- 匹配队列
- 排行榜
- 赛季奖励结算

我没有把它做成大而全的商业服，而是先把“入队、撮合、对局回写、Redis 热榜查询、赛季结算、奖励领取防重复”这条链路做通。

## 为什么这个项目适合 4399

因为它补的是我现有项目里还比较缺的一条线：

- `cpp-room-server` 更偏房间、广播、心跳、重连
- 这个项目更偏匹配、排行、Redis 热榜、结算、奖励和持久化

组合起来以后，更像游戏服务端里不同模块的拆分。

## 你在项目里最想讲的点

### 1. 匹配为什么用等待时间扩圈

一开始玩家只在较小分差内匹配，保证公平性；
等得越久，窗口越大，避免一直匹配不到。

当前实现是：

- 基础窗口 50 分
- 每秒增加 80 分

这不是最终商用算法，但很适合说明我对“公平性和等待时长取舍”的理解。

### 2. 为什么奖励领取要防重复

因为奖励通常是“可查询、可领取”的状态型数据。
如果只看接口调用，不做领取状态控制，就容易出现重复发奖。

我这版是把 `claimed` 和 `claimed_at` 放进数据库，通过更新条件保证同一条奖励只能成功领取一次。

### 3. 为什么排行榜要加 Redis

因为这个项目里最适合做热数据分层的就是排行榜：

- `TOP N` 查询频繁
- 个人排名查询天然适合缓存
- Redis `ZSET` 可以直接按分数排序

当前设计是：

- SQLite 存玩家真实积分和对局结果
- Redis 只做排行榜热数据
- 启动时支持从 SQLite 重建 Redis 榜单

这样我在面试里就能把“持久化”和“热数据”分层说清楚。

### 4. 为什么对局完成要一起更新

提交对局结果后，至少会同时影响三类状态：

- 对局本身从 pending 变成 finished
- 胜者积分变化
- 负者积分变化

所以我把这几步放进一个数据库事务里，避免出现“对局已完成，但积分没改完”这种半成功状态。
